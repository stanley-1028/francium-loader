## 📊 現狀分析

### 目前的支援程度

| 功能 | 狀態 | 說明 |
|------|------|------|
| Forge 模組載入 | ⚠️ 基本支援 | 可以載入簡單的 Forge 模組 |
| FML 生命週期 | 🚧 部分實現 | 基本的生命週期事件 |
| Forge 註冊系統 | 🚧 部分實現 | 基本的註冊功能 |
| Forge 事件系統 | ❌ 未實現 | 還沒有完整的事件匯流排 |
| Forge 配置系統 | ❌ 未實現 | 還不支援 Forge 配置 |
| Forge 能量系統 | ❌ 未實現 | FE/RF 能量系統 |
| Forge 流體系統 | ❌ 未實現 | 流體 API |
| Forge 能力系統 | ❌ 未實現 | Capability 系統 |

### 已有的基礎

1. **雙生態架構設計**
   - Francium 的架構本來就設計成支援多種模組格式
   - 有基本的模組格式偵測和分派機制

2. **Mixin 整合**
   - 已經內建 SpongePowered Mixin
   - Forge 也使用 Mixin，這部分可以共用

3. **AI Bridge 技術**
   - 可以用 AI Bridge 來輔助 Forge API 的適配
   - 自動生成部分轉換程式碼

### 為什麼需要 Forge 支援？

1. **生態完整性**
   - Forge 有非常龐大的模組生態
   - 很多經典模組只有 Forge 版本
   - 完整的雙生態支援是 Francium 的核心賣點之一

2. **使用者需求**
   - 很多玩家同時使用 Forge 和 Fabric 模組
   - 目前需要在兩個加載器之間切換，很不方便
   - Francium 可以解決這個痛點

3. **技術挑戰**
   - Forge 的 API 比較複雜，實現起來有挑戰性
   - 但也是展示 Francium 技術實力的機會

## 🎯 目標與範圍

### 總體目標

實現對 Forge 模組的完整支援，讓絕大多數 Forge 模組可以在 Francium 上正常執行。

### 具體目標

#### 短期目標（v2.5 - v2.6）

- ✅ 能夠載入大多數簡單的 Forge 模組
- ✅ 基本的 FML 生命週期支援
- ✅ 基本的註冊系統支援
- ✅ 基本的事件系統支援
- ✅ 能夠執行一些受歡迎的 Forge 模組

#### 中期目標（v3.0）

- ✅ 完整的 Forge 事件系統
- ✅ 完整的註冊系統
- ✅ Forge 配置系統支援
- ✅ 能量系統（FE/RF）支援
- ✅ 流體系統支援
- ✅ 能力（Capability）系統支援
- ✅ 大部分 Forge 模組可以正常執行

#### 長期目標（v3.5+）

- ✅ 近乎完整的 Forge API 覆蓋
- ✅ Forge 和 Fabric 模組的深度整合
- ✅ 跨加載器的模組交互
- ✅ 效能優化
- ✅ 完整的除錯和開發工具

### 不在範圍內的項目

1. **Forge 模組開發工具鏈**
   - 不提供 Forge 的 Gradle 外掛
   - 不提供 Forge 的開發環境
   - 模組開發者仍然使用官方工具鏈

2. **Forge 特定的優化**
   - 不實現 Forge 的性能優化
   - 使用者可以使用其他優化模組

3. **舊版 Forge 支援**
   - 主要支援較新的 Forge 版本（1.20+）
   - 舊版本可能不會特別最佳化

## 🔧 技術方案

### 整體架構

```
┌─────────────────────────────────────────────────────┐
│                   Francium Core                      │
├─────────────────────────────────────────────────────┤
│  Mod Discovery / Loading / DAG / SAT Solver         │
└──────────────┬──────────────────────┬───────────────┘
               │                      │
    ┌──────────▼──────────┐  ┌────────▼──────────┐
    │  Fabric Adapter     │  │  Forge Adapter    │
    │  (已實現)           │  │  (開發中)         │
    └──────────┬──────────┘  └────────┬──────────┘
               │                      │
    ┌──────────▼──────────┐  ┌────────▼──────────┐
    │  Fabric Mods        │  │  Forge Mods        │
    └─────────────────────┘  └───────────────────┘
```

### Forge 適配層的核心模組

#### 1. FML 模擬層（FML Emulation Layer）

**作用：** 模擬 Forge Mod Loader (FML) 的環境

**主要功能：**
- FML 生命週期事件
- 模組載入和管理
- 依賴解析
- 版本檢查

**關鍵類別：**
- `FMLModContainer` - Forge 模組容器
- `FMLLifecycle` - FML 生命週期管理
- `FMLModLoadingContext` - 模組載入上下文

#### 2. 事件匯流排（Event Bus）

**作用：** 實現 Forge 的事件系統

**主要功能：**
- 事件發布和訂閱
- 事件優先級
- 事件取消機制
- 各種遊戲事件

**關鍵類別：**
- `ForgeEventBus` - 事件匯流排實現
- `ForgeEvent` - 事件基類
- 各種具體事件類

#### 3. 註冊系統（Registry System）

**作用：** 實現 Forge 的註冊系統

**主要功能：**
- 物件註冊
- 註冊事件
- RegistryBuilder
- DeferredRegister

**關鍵類別：**
- `ForgeRegistry` - Forge 註冊表
- `DeferredRegister` - 延遲註冊
- `RegistryEvent` - 註冊事件

#### 4. 配置系統（Config System）

**作用：** 實現 Forge 的配置系統

**主要功能：**
- 配置檔案讀寫
- 配置類型（客戶端/伺服器/通用）
- 配置訂閱和重載
- 配置 GUI 支援

**關鍵類別：**
- `ForgeConfig` - 配置管理
- `ConfigSpec` - 配置規範
- `ModConfig` - 模組配置

#### 5. 能量系統（Energy System）

**作用：** 實現 Forge Energy (FE/RF) 系統

**主要功能：**
- IEnergyStorage 介面
- 能量傳輸
- 能量單元物品

**關鍵類別：**
- `EnergyStorage` - 能量儲存實現
- `EnergyCapability` - 能量能力提供者

#### 6. 流體系統（Fluid System）

**作用：** 實現 Forge 的流體 API

**主要功能：**
- 流體註冊
- 流體屬性
- 流體交互
- 流體容器

**關鍵類別：**
- `ForgeFluid` - 流體實現
- `FluidAttributes` - 流體屬性

#### 7. 能力系統（Capability System）

**作用：** 實現 Forge 的 Capability 系統

**主要功能：**
- 能力提供者
- 能力附加
- 能力序列化
- 能力事件

**關鍵類別：**
- `CapabilityManager` - 能力管理器
- `CapabilityProvider` - 能力提供者

### AI Bridge 的角色

AI Bridge 在 Forge 適配中扮演重要角色：

1. **API 對應自動生成**
   - 自動分析 Forge API 和 Minecraft API 的對應關係
   - 生成轉換層程式碼

2. **方法匹配**
   - 幫助找到 Forge 方法在 Francium 中的對應實現
   - 提高適配層的開發效率

3. **跨版本相容**
   - 不同版本的 Forge API 可能有差異
   - AI Bridge 可以幫助處理這些差異

## 📅 分階段實現計劃

### 第一階段：基礎架構（v2.5）

**目標：** 建立 Forge 適配層的基礎架構，能夠載入簡單的 Forge 模組

**工作項目：**

1. **Forge 模組格式偵測**
   - 辨識 Forge 模組的 JAR 檔案
   - 讀取 mcmod.info / mods.toml
   - 提取模組中繼資料

2. **FML 生命週期基本實現**
   - 基本的模組載入流程
   - 主要的生命週期事件
   - 模組類別實例化

3. **基本註冊系統**
   - 基本的 Registry 實現
   - 支援方塊、物品、實體等基本註冊
   - 註冊事件觸發

4. **Mixin 整合**
   - 確認 Forge 的 Mixin 可以正常載入
   - 處理 Mixin 配置
   - 解決 Mixin 衝突

5. **測試模組**
   - 選擇幾個簡單的 Forge 模組進行測試
   - 驗證基本功能
   - 記錄問題和瓶頸

**預期成果：**
- 能夠載入 5-10 個簡單的 Forge 模組
- 基本的遊戲功能正常
- 有明確的問題清單和下一步計劃

**工作量估計：** 2-3 週

---

### 第二階段：事件與配置（v2.6）

**目標：** 實現事件系統和配置系統，支援更多 Forge 模組

**工作項目：**

1. **完整的事件匯流排**
   - 實現 IEventBus 介面
   - 支援事件優先級
   - 支援事件取消
   - 實現主要的遊戲事件

2. **Forge 配置系統**
   - 實現 ForgeConfigSpec
   - 支援三種配置類型
   - 配置自動儲存和載入
   - 配置重載事件

3. **擴充註冊系統**
   - 支援更多註冊類型
   - 實現 DeferredRegister
   - 支援 RegistryBuilder

4. **模組通訊（Networking）**
   - 基本的網路封包系統
   - 簡單的封包處理
   - 用戶端-伺服端通訊

5. **更多測試**
   - 測試更多 Forge 模組
   - 包括一些中等複雜度的
   - 性能基準測試

**預期成果：**
- 能夠載入 20-30 個 Forge 模組
- 大部分中等複雜度的模組可以執行
- 事件和配置系統基本可用

**工作量估計：** 3-4 週

## ⚠️ 風險與挑戰

### 技術風險

#### 1. Forge API 的複雜性
**風險：** 高
**說明：** Forge 的 API 非常龐大且複雜，完整實現需要大量工作。
**緩解措施：**
- 分階段實現，先做最常用的部分
- 優先實現受歡迎模組使用的 API
- 使用 AI Bridge 輔助自動生成

#### 2. 版本差異
**風險：** 中高
**說明：** 不同版本的 Forge API 可能有差異，需要處理相容性。
**緩解措施：**
- 優先支援最新的穩定版本
- 使用 AI Bridge 處理版本差異
- 提供版本相容性層

#### 3. 效能問題
**風險：** 中
**說明：** 適配層可能會帶來效能開銷。
**緩解措施：**
- 精心設計架構，減少間接層
- 效能基準測試和最佳化
- 提供效能設定選項

#### 4. Mixin 衝突
**風險：** 中
**說明：** Forge 模組的 Mixin 可能和 Francium 或其他模組衝突。
**緩解措施：**
- 完善的 Mixin 衝突檢測
- 提供 Mixin 管理工具
- 文件和最佳實踐指南

### 專案風險

#### 1. 工作量龐大
**風險：** 高
**說明：** 完整的 Forge 支援是一個很大的專案，需要很多時間和資源。
**緩解措施：**
- 分階段交付，每個階段都有可用的成果
- 優先實現最有價值的功能
- 歡迎社群貢獻

#### 2. 維護負擔
**風險：** 中高
**說明：** Forge 經常更新，需要持續維護適配層。
**緩解措施：**
- 自動化測試，減少迴歸
- 模組化設計，易於更新
- 文件完善，降低參與門檻

### 生態風險

#### 1. 模組開發者接受度
**風險：** 中
**說明：** 模組開發者可能不願意針對 Francium 調整他們的模組。
**緩解措施：**
- 目標是「開箱即用」，不需要模組作者修改
- 提供除錯工具，幫助模組作者解決問題
- 積極與模組開發者溝通

#### 2. 與 Forge 的關係
**風險：** 低
**說明：** 需要確保不會侵犯 Forge 的權益。
**緩解措施：**
- 遵循 Forge 的授權條款
- 不包含 Forge 的原始碼
- 明確說明這是獨立的實現
